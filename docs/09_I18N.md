## 1. Âmbito

Este manual descreve o sistema de internacionalização (i18n) da app AFROLOC:

- os **13 idiomas suportados** (com nomes completos e códigos);
- a **estrutura das chaves de tradução** e o formato dos ficheiros;
- como a função `t()` é usada na aplicação (contexto `LanguageContext`);
- o **processo de validação** de traduções (ferramenta de admin + validador);
- um **problema conhecido** relevante: valores de ENUM vindos da base de dados (ex.: severidade `low`/`medium`/`critical`, tipos de evento `otp_request`/`phone_change`) mostrados **crus** no Monitor de Segurança em vez de passarem por `t()`;
- boas práticas para autores de código e de conteúdo.

Ficheiros de referência (âncoras deste manual):

- `src/contexts/LanguageContext.tsx` — provider, tipo `Language`, função `t()`, carregamento dinâmico e cascata de fallback.
- `src/translations/*.json` — 13 dicionários planos chave→valor.
- `src/utils/translationValidator.ts` — lógica de validação, estatísticas, export/import Excel.
- `src/components/TranslationValidator.tsx` — UI da ferramenta de validação (admin).
- `src/components/LanguageSelector.tsx` — seletor de idioma na UI.
- `src/pages/AdminAutoTranslate.tsx` — página de tradução automática (admin).
- `supabase/functions/translate-keys/index.ts` — Edge Function de tradução automática por IA.
- `public/TRANSLATION_VALIDATION.md` — documentação da ferramenta de validação.

## 2. Idiomas suportados

A app suporta **13 idiomas**. O tipo canónico está declarado em `src/contexts/LanguageContext.tsx` (e replicado em `src/utils/translationValidator.ts`):

```ts
export type Language = 'en' | 'fr' | 'pt' | 'ar' | 'am' | 'sw' | 'ln' | 'yo' | 'sn' | 'zu' | 'kmb' | 'umb' | 'kg';
```

Cada código corresponde a um ficheiro `src/translations/<código>.json` (confirmados: `am, ar, en, fr, kg, kmb, ln, pt, sn, sw, umb, yo, zu`).

| Código | Nome nativo (na UI de validação) | Nome (inglês) | Notas |
|---|---|---|---|
| `pt` | Português | Portuguese | **Idioma-base / baseline** e mercado principal |
| `en` | English | English | Idioma de **referência** e fallback secundário |
| `fr` | Français | French | |
| `ar` | العربية (Arabic) | Arabic | Escrita da direita para a esquerda (RTL) — ⚠️ a validar se há tratamento RTL dedicado |
| `am` | አማርኛ (Amharic) | Amharic | |
| `sw` | Kiswahili | Swahili | |
| `ln` | Lingála | Lingala | |
| `yo` | Yorùbá | Yoruba | |
| `sn` | chiShona | Shona | |
| `zu` | isiZulu | Zulu | |
| `kmb` | Kimbundu | Kimbundu | Língua nacional de Angola |
| `umb` | Umbundu | Umbundu | Língua nacional de Angola |

Os nomes nativos usados na interface vêm do mapa `languageNames` em `src/utils/translationValidator.ts`. A lista de nomes em inglês usada pela tradução automática vive em `supabase/functions/translate-keys/index.ts` (mapa `languageNames`).

**Idioma por omissão:** `pt`. É lido de `localStorage` (chave `afroloc-language`) no arranque; se não existir, assume `pt`.

## 3. Estrutura das chaves e uso de `t()`

### 3.1 Formato dos ficheiros de tradução

Cada `src/translations/<lang>.json` é um objeto **plano** (não aninhado) de `Record<string, string>` — chave textual → valor traduzido. Exemplo (`pt.json`):

```json
{
  "app_name": "AFROLOC",
  "welcome": "Bem-vindo",
  "ready_for_verification": "Pronto para Verificação",
  "ready_for_verification_description": "Você tem {count} testemunhas confirmadas. O seu AFROLOC já pode ser verificado."
}
```

- As chaves são identificadores estáveis (snake_case), **não** o texto humano.
- Os valores podem conter **placeholders** entre chavetas: `{count}`, `{days}`, `{hours}`, `{name}`, etc. Estes são preservados literalmente pela tradução automática (ver 4.2) e substituídos em runtime pela app (tipicamente via `.replace("{days}", …)`, ex. `src/pages/ChangePhone.tsx`).
- Contagem atual de chaves (referência): **`pt` ≈ 4807**, **`en` ≈ 4636** (`pt` é o baseline e tem mais chaves).

### 3.2 O provider e a função `t()`

`LanguageProvider` (em `src/contexts/LanguageContext.tsx`) expõe o contexto:

```ts
interface LanguageContextType {
  language: Language;
  setLanguage: (lang: Language) => void;
  t: (key: string) => string;
}
```

**Carregamento dinâmico (performance):** apenas `pt` e `en` são importados estaticamente e ficam no bundle principal — servem simultaneamente de mercado principal e de fallback, por isso há sempre tradução imediata (zero flash). As outras **11 línguas carregam por chunks separados** (`import(...)`) só quando escolhidas, poupando ~1,4 MB no primeiro carregamento (crítico para telemóveis modestos / dados limitados).

**Cascata de fallback** da função `t()`:

```ts
const t = (key: string): string => (active && active[key]) || en[key] || pt[key] || key;
```

Ou seja, por ordem: **idioma ativo → inglês → português → a própria chave**. Se uma chave não existir em lado nenhum, é devolvida a chave crua (útil para detetar em dev chaves em falta).

**Efeitos:** `setLanguage(lang)` grava em `localStorage` (`afroloc-language`) e define `document.documentElement.lang`. Um `useEffect` mantém o `lang` do `<html>` sincronizado e faz o import dinâmico da língua ativa se ainda não estiver em memória.

**Resiliência a HMR:** se `useLanguage()` for chamado fora do provider (pode acontecer durante hot-reload), devolve um fallback seguro (`language: 'pt'`, `t: (key) => key`) e regista um `console.warn` em vez de crashar.

### 3.3 Como usar na app

Regra geral: **nunca escrever texto humano fixo no JSX** — usar sempre `t('chave')`.

```tsx
import { useLanguage } from '@/contexts/LanguageContext';

function MinhaComponente() {
  const { t, language, setLanguage } = useLanguage();
  return <h1>{t('welcome')}</h1>;
}
```

`useLanguage` é usado em ~137 ficheiros de `src` (referência). O seletor de idioma na UI é `src/components/LanguageSelector.tsx`.

**Placeholders:** como `t()` devolve a string bruta, a interpolação é feita manualmente no local de uso, por exemplo:

```tsx
t('phone_change_cooldown_message').replace('{days}', String(data.days_remaining))
```

## 4. Processo de validação

### 4.1 Ferramenta de validação (admin)

Documentada em `public/TRANSLATION_VALIDATION.md`; lógica em `src/utils/translationValidator.ts`; UI em `src/components/TranslationValidator.tsx`.

Acesso: **Admin → Translation Validation** (requer privilégios de admin).

Funções principais (`src/utils/translationValidator.ts`):

- `getAllTranslationKeys()` — reúne o conjunto (`Set`) de **todas** as chaves únicas existentes em todos os idiomas.
- `findMissingTranslations(language, allKeys)` — devolve as chaves em falta num idioma, com o valor de referência em inglês (`translations.en[key]`).
- `calculateCompletionRate(language, totalKeys)` — percentagem de chaves preenchidas (valores não vazios após `trim()`), arredondada.
- `validateTranslations()` — gera o relatório completo (`ValidationReport`), com `languageStats` ordenadas por taxa de conclusão. O **baseline reportado é `pt`** (`baselineLanguage: 'pt'`).
- `printValidationReport(report)` — imprime relatório formatado na consola (útil em dev).
- `exportReportToJSON(report)` / `exportTranslationsToExcel()` — exportação para JSON e para Excel (`.xlsx`, uma folha `Translations` com colunas por idioma).
- `importTranslationsFromExcel(file)` / `generateTranslationFiles(...)` — importação a partir de Excel e geração dos ficheiros JSON por idioma (mapeando cabeçalhos como `Português`, `English`, `العربية`, etc. para códigos).

> ⚠️ Nota sobre a taxa de conclusão: `calculateCompletionRate` usa como denominador o **total de chaves únicas em todos os idiomas** (que pode exceder o baseline `pt`), pelo que a percentagem é uma métrica de cobertura global, não estritamente "% do baseline".

Em **modo de desenvolvimento** (`import.meta.env.DEV`), a ferramenta pode validar no carregamento, registar avisos na consola e mostrar um botão de estado flutuante (ver `public/TRANSLATION_VALIDATION.md`). Há ainda `trackTranslationUsage(key)` / `getMissingKeysReport()` para detetar, em runtime de dev, chaves usadas mas em falta.

### 4.2 Tradução automática por IA (admin)

Página: `src/pages/AdminAutoTranslate.tsx`. Edge Function: `supabase/functions/translate-keys/index.ts`.

Fluxo:

1. A página calcula, por idioma, as chaves em falta ou ainda iguais ao valor `pt` (isto é, não traduzidas), usando `pt.json` como fonte.
2. Invoca `supabase.functions.invoke('translate-keys', { body: { sourceLanguage: 'pt', targetLanguage: lang, keys } })`.
3. A Edge Function traduz **de Português** para o idioma-alvo, em lotes de 50 chaves, chamando um AI gateway (modelo `google/gemini-2.5-flash` — ⚠️ a validar se se mantém). Regras impostas no prompt: preservar placeholders (`{count}`, `{days}`, …); manter os nomes das chaves; manter o tom; manter termos técnicos em inglês (`AFROLOC`, `GPS`, `OTP`, `API`); devolver apenas JSON válido.
4. Trata `429` (rate limit) devolvendo resultado parcial; introduz pausa de 500 ms entre lotes.
5. A página permite descarregar o `<lang>.json` resultante para revisão/commit manual.

> A tradução automática é um **acelerador**, não um substituto de revisão humana. O resultado deve ser revisto antes de ser commitado, sobretudo para as línguas nacionais angolanas.

### 4.3 Adicionar um novo idioma

1. Criar `src/translations/<código>.json`.
2. Acrescentar o código ao tipo `Language` (em `LanguageContext.tsx` **e** em `translationValidator.ts`).
3. Registar o loader dinâmico em `loaders` (`LanguageContext.tsx`) e a entrada em `translations` (`translationValidator.ts`).
4. Acrescentar o nome nativo em `languageNames` (validador) e o nome inglês em `translate-keys/index.ts`.
5. Atualizar o `LanguageSelector`.

## 5. Problema conhecido: enums crus (não passam por `t()`)

**Ficheiro:** `src/pages/SecurityMonitoring.tsx` (Monitor de Segurança).

Alguns valores de ENUM vindos da base de dados são mostrados **crus** ao utilizador, em vez de passarem por `t()`. Isto quebra a internacionalização (aparece em inglês/técnico independentemente do idioma) e expõe strings de sistema.

### 5.1 Severidade renderizada com `.toUpperCase()`

Nas linhas ~353 e ~474 a severidade é mostrada assim:

```tsx
<Badge className={getSeverityColor(event.severity)}>
  {event.severity.toUpperCase()}
</Badge>
```

O campo `event.severity` (`low` / `medium` / `high` / `critical`, tipado como `string`) é apenas passado por `.toUpperCase()` → aparece **`LOW`**, **`MEDIUM`**, **`CRITICAL`** em maiúsculas, sem tradução.

### 5.2 Tipos de evento com mapa incompleto

`getEventTypeLabel` (linha ~156) só mapeia **4** tipos por `t()`:

```tsx
const getEventTypeLabel = (type: string) => {
  const typeMap: Record<string, string> = {
    rate_limit: t("rate_limit"),
    auth_failure: t("auth_failure"),
    suspicious_activity: t("suspicious_activity"),
    brute_force: t("brute_force")
  };
  return typeMap[type] || type;   // fallback: devolve o enum cru
};
```

Qualquer outro tipo de evento — por exemplo **`otp_request`** ou **`phone_change`** — cai no fallback `|| type` e é mostrado **cru** (o identificador técnico, não traduzido nem humanizado).

### 5.3 Padrão de correção recomendado

Aplicar o mesmo princípio a todos os enums vindos da BD: **mapa de tradução por enum + fallback humanizado** (nunca o valor cru).

1. Adicionar chaves de tradução em todos os JSON, com prefixo por domínio, ex.:
   - severidade: `severity_low`, `severity_medium`, `severity_high`, `severity_critical`;
   - tipo de evento: `event_type_otp_request`, `event_type_phone_change`, `event_type_rate_limit`, `event_type_auth_failure`, `event_type_suspicious_activity`, `event_type_brute_force`.
2. Substituir a renderização crua por um helper com fallback **humanizado** (substituir `_` por espaço + capitalizar), nunca `.toUpperCase()` do valor bruto nem `|| type`:

   ```tsx
   const humanize = (v: string) =>
     v.replace(/_/g, ' ').replace(/\b\w/g, c => c.toUpperCase());

   const labelSeverity = (s: string) => {
     const key = `severity_${s}`;
     const translated = t(key);
     return translated === key ? humanize(s) : translated; // t() devolve a chave se faltar
   };

   const labelEventType = (type: string) => {
     const key = `event_type_${type}`;
     const translated = t(key);
     return translated === key ? humanize(type) : translated;
   };
   ```

   ```tsx
   <Badge className={getSeverityColor(event.severity)}>{labelSeverity(event.severity)}</Badge>
   ...
   {labelEventType(event.event_type)}
   ```

   O teste `translated === key` funciona porque a cascata de `t()` devolve a própria chave quando não há tradução (ver 3.2) — permitindo cair no fallback humanizado em vez de mostrar `SEVERITY_LOW` cru.

3. Fazer o mesmo varrimento a outros pontos onde valores de BD sejam mostrados via `.toUpperCase()`, `.replace(...)` ad-hoc ou mapas incompletos. ⚠️ A validar: inventariar todos os enums da BD renderizados na UI (severidade, tipos de evento, estados, papéis, etc.).

## 6. Boas práticas

**Para quem escreve código:**

1. Nunca escrever texto humano fixo no JSX — usar sempre `t('chave')`.
2. Adicionar a chave primeiro a **`pt.json`** (baseline) e a `en.json` (referência), e só depois às restantes.
3. Usar chaves estáveis em snake_case; agrupar por prefixo de domínio (ex.: `autotrans_*`, `phone_change_*`, `severity_*`).
4. Manter placeholders entre chavetas (`{count}`) e interpolar no local de uso; não concatenar strings traduzidas manualmente por causa da ordem das palavras entre línguas.
5. **Enums vindos da BD nunca são texto de UI** — mapear sempre por `t()` com fallback humanizado (ver secção 5), nunca `.toUpperCase()` do valor cru.

**Para quem gere conteúdo/traduções:**

6. `pt` é o baseline e `en` a referência; manter ambos completos.
7. Prioridade de conclusão (de `public/TRANSLATION_VALIDATION.md`): `en` (referência) → `pt` (base de utilizadores) → `fr` → `sw`; objetivo ≥ 90% para línguas de produção.
8. Correr a validação (Admin → Translation Validation) antes de releases; exportar o relatório JSON/Excel para acompanhamento.
9. Rever sempre a tradução automática antes do commit — em especial `kmb`, `umb`, `kg` (línguas nacionais angolanas).
10. Manter termos técnicos em inglês (`AFROLOC`, `GPS`, `OTP`, `API`), conforme já imposto no prompt da Edge Function.
